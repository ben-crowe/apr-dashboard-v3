## Template Rendering Architecture

**Function**: `renderPhotosSection()` in `reportHtmlTemplate.ts:3216-3330`

**Rendering Logic**:
1. Loop through each subsection (EXTERIOR, STREET, COMMON, UNIT, SYSTEMS)
2. Extract image fields from subsection.fields where `field.type === 'image'`
3. Find matching caption field by appending `-caption` to image field ID
4. Group into `{ imageUrl, caption }` objects
5. Render in 2-column grid layout
6. Display subsection title (e.g., "EXTERIOR PHOTOGRAPHS")
7. Add page break every 2 rows (4 photos per page)

**CSS Grid Layout**:
```css
display: grid;
grid-template-columns: repeat(2, 1fr);
gap: 20px;
```

## Storage Architecture Notes

**Current State (TDD)**: 
- All images stored as Base64 data URLs
- Embedded directly in Zustand store state
- No external storage required
- Status shown as "→ Local (data URL)" in TDD

**Future State (Production)**:
- Images will upload to Supabase Storage
- Stored in `documents` bucket (production) or `job-files` bucket (V3 Dashboard)
- Fields will contain HTTP URLs instead of Base64 strings
- Template rendering logic supports both data URLs and HTTP URLs

**Template Compatibility**:
The `renderPhotosSection()` function handles both string values and array values:

```typescript
let imageUrl: string | null = null;
if (typeof field.value === 'string' && field.value) {
  imageUrl = field.value;  // Works for both data URLs and HTTP URLs
} else if (Array.isArray(field.value) && field.value.length > 0) {
  imageUrl = field.value[0] as string;
}
```

This ensures seamless transition from data URLs to Supabase Storage URLs.
