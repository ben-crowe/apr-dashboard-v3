## 1 — What it is

**GOAL:** At the moment a job enters the system, a matching SharePoint folder set is created automatically — parent folder named by job number + property description, plus 5 locked subfolders — so the team has an immediate, structured home for the report, client files, work files, billing documents, and the letter of reliance. This is the per-job document home that the whole team uses.

**Phase sequence:** follows Phase 3 (Valcre job creation). The SharePoint folder-create is a third step in the same on-intake automation chain: create Valcre job → create ClickUp card → create SharePoint folder set. It also has an on-demand path: a "Create Client Folders" button in the Stage-2 job detail for cases where auto-creation didn't fire or the appraiser wants to confirm folders are connected.

**Current state:** NOT built. The Entra app registration that powers the Graph call does not exist yet — it is the first item in the to-build cluster. No folder-create code, no dashboard section for it.

## 3 — The fields / data

**GOAL:** Every job record carries the SharePoint parent folder URL once folders are created, and the 5 subfolder paths are derivable deterministically from it.

**The locked folder spec (verbatim — casing and ` - ` separator are intentional):**

**Parent folder name:** `{JOB#} - {Property Description + Street Address + City + Province}`

Example live folder: `VAL261054 - Stacked Townhouse Development 2822 &2828 11 Ave SE Calgary AB`

**Parent path:** `.../sites/V/Shared Documents/2.Jobs/{YEAR}/{PARENT_FOLDER_NAME}/`

**5 subfolders (order and naming locked):**

| # | Subfolder name |
|---|---|
| 1 | `1. REPORT` |
| 2 | `2. CLIENT SUPPLIED` |
| 3 | `3. WORK FILES (TTSZ, PICS, COMPS)` |
| 4 | `4. CLIENT BILLING (Invoice, LOE)` |
| 5 | `5. LETTER OF RELIANCE (LOR)` |

**Fields the app already holds that build the path:**

| Field | Source | Notes |
|---|---|---|
| `{JOB#}` | Valcre job number (e.g. `VAL261054`) | Available post Phase 3 |
| `{YEAR}` | Derived from job number (`VAL26…` → `2026`) or job created date | Derivable without extra call |
| `{Property Description}` | The job name generated by the APR app | Must match exactly what Valta uses on disk |

**DB field to add:** `sharepoint_folder_url` (or `sharepoint_folder_connected_at` + URL) on `job_loe_details` — persists the parent folder URL once created, used by the dashboard section to show connected state and open-in-SharePoint links.

## 4 — Data flow in → out

**GOAL:** Job-create fires → Graph call out → folder set exists in SharePoint → URL lands on the job record → dashboard shows "Connected."

**In (from prior phases):**

- Valcre job number (Phase 3) — the `{JOB#}` component
- Property description / job name (Phase 1 submission data) — the title component
- Job year — derived from job number or created date

**The call (to-build):**

```
POST /api/sharepoint-folders
  body: { jobId, jobNumber, jobName, year }
  → Graph: check-then-create parent folder
  → Graph: check-then-create each of the 5 subfolders
  → return: { parentFolderUrl, subfolders[] }
```

**Out (to the job record):**

- `sharepoint_folder_url` written to `job_loe_details` (or equivalent)
- Dashboard ClientFoldersSection reads this to render connected state + open-in-SharePoint links

**Downstream phases that use the folders:**

- Signed LOE PDF (Phase 5 completion) → pushed to `4. CLIENT BILLING (Invoice, LOE)`
- Client-supplied attachments → routed to `2. CLIENT SUPPLIED`
- Final report → saved to `1. REPORT`
- LOE / LOR → saved to `5. LETTER OF RELIANCE (LOR)`
- File naming convention for the signed LOE is TBD — Ben to confirm.

## 5 — Trigger / gating logic

**GOAL:** Folders exist in SharePoint the moment a job is in the system, with no manual step required. An on-demand button handles missed cases and gives appraisers a clear "Connected" confirmation.

---

### Trigger Option A — Auto at job creation (recommended default)

The Graph folder-create call joins the existing intake chain as a third step:

```
Job received in APR app
  → create Valcre job        (Phase 3 — exists)
  → create ClickUp card      (Phase 3 — exists)
  → create SharePoint folder set  ← NEW, fires here
```

**Pro:** team immediately has a folder home; no manual action ever needed; the ClickUp card + folder set arrive together.

**Con:** if Graph creds are missing or the call fails, job creation must not fail — this step must be fire-and-forget with error logged, not blocking. Also means the folder is created even for quotes that never convert to a paid job (wasted folders if conversion rate is low).

---

### Trigger Option B — Appraiser presses "Create Client Folders" button

A button in the Stage-2 job detail fires the same Graph call on demand.

**Pro:** appraiser decides when the job is real enough to warrant a folder; avoids folders for dead quotes; gives a tactile confirmation moment.

**Con:** manual step; easy to forget; client folders may not exist when an appraiser needs them during field work.

**Recommendation (not picked):** present both to Ben. Auto at job-create (Option A) is the natural fit for the rest of the intake chain. The button (Option B) should be built regardless — it is the fallback for re-connect and for jobs created before the auto-create existed.

---

### Two timing options (client preference unknown — present both)

**Option T1 — Create folders on quote / LOE send (before signing)**

Folders are created when the LOE goes out (Phase 5 trigger), or at job creation if auto-create is live.

- Pro: team can start collecting work files and client-supplied docs immediately, even before the engagement is signed.
- Con: folder set exists for jobs the client never signs. If the client declines, the folder stays in SharePoint as a ghost.

**Option T2 — Create folders after LOE signing**

Folders are not created until `loe_signed_at` is stamped (Phase 5 completion webhook).

- Pro: only committed, signed engagements get a folder — clean SharePoint library.
- Con: there is a gap between LOE send and signing (days to weeks) where the team has no folder home; work files and client-supplied docs may accumulate in ad-hoc locations.

Ben to choose. Both are a config of the same underlying trigger — the Graph call is identical either way.

---

### Button states (to-build, from the recipe)

| State | UI label | Condition |
|---|---|---|
| Not yet created | `Create Client Folders` | No `sharepoint_folder_url` on the job |
| In progress | `Creating…` (spinner) | Graph call in flight |
| Connected | `Folders Connected ✓` + "Open in SharePoint" link | `sharepoint_folder_url` populated |
| No creds | `SharePoint not configured` (disabled) | Entra creds not present server-side |

**Idempotent behavior:** clicking "Create Client Folders" when folders already exist does NOT duplicate them. The Graph call is check-then-create per folder (GET → only POST if 404). So "make folders" and "re-connect folders" are the same safe action.

## 6 — User + agent flow

**GOAL:** Appraiser opens the job detail, sees "Folders Connected ✓" with a link straight into SharePoint — no manual folder creation, no clipboard-paste of paths.

**Human path (button flow):**

1. Open job detail for a job with a Valcre job number.
2. Scroll to the Client Folders section (last section, below Document Upload & Organization).
3. If not yet connected: press "Create Client Folders."
4. Section shows "Creating…" while the Graph call runs.
5. On success: section flips to "Folders Connected ✓" with the parent folder open-in-SharePoint link and the 5 subfolder links listed.
6. Press "Open in SharePoint" to navigate directly to the job folder.

**Human path (auto-create, once wired):**

The section arrives already showing "Folders Connected ✓" — the intake chain created the folder set automatically. No button press needed.

**Agent test path:**

- Verify `sharepoint_folder_url` is null on a test job, then call `POST /api/sharepoint-folders` directly with the job's data, then confirm the field is populated and the folder exists via a Graph `GET` check.
- For the button: use `/cli-browser-auto` to load the job detail, take a screenshot of the Client Folders section, click "Create Client Folders," screenshot the resulting connected state, then verify `sharepoint_folder_url` in Supabase.
- For auto-create: trigger job creation end-to-end and confirm the folder was created without any button press.

### 2. Folder-create Graph API endpoint (co-architect + react-specialist)

New Vercel function `api/sharepoint-folders.ts`:

- Acquires token via client-credentials flow (tenant ID + client ID + secret → `POST https://login.microsoftonline.com/{tenantId}/oauth2/v2.0/token`)
- Resolves the SharePoint site drive ID once and caches (or hardcodes after first discovery)
- Creates parent folder: `POST /drives/{driveId}/root:/2.Jobs/{YEAR}/{PARENT_FOLDER_NAME}:/children` (check-then-create per folder — idempotent)
- Creates each of the 5 subfolders under the parent (check-then-create)
- Returns parent folder URL + subfolder URLs
- Error must NOT propagate to block job creation — fire-and-forget with logged error

**Owner:** co-architect (wires the Graph call) — react-specialist connects the button to it.

### 4. Folder name field-match: rename dashboard upload categories to match SharePoint (react-specialist)

The current Section 4 upload UI uses document-type labels that do NOT correspond to SharePoint subfolder names. This is expected — Section 4 is about specific document types uploaded to Supabase buckets, while SharePoint subfolders are the team's file home. They are parallel systems, not the same UI.

However, when files are eventually pushed FROM Supabase/the app INTO SharePoint (signed LOE, client attachments, final report), the routing logic must map to the exact subfolder names below. This is a backend routing concern, not a UI rename.

**Field-match check — dashboard upload labels vs SharePoint subfolder names:**

| SharePoint subfolder (locked, verbatim) | Current dashboard upload label | Match? |
|---|---|---|
| `1. REPORT` | *(no equivalent upload type)* | No upload type — final report is generated, not uploaded |
| `2. CLIENT SUPPLIED` | *(no equivalent)* | No direct equivalent — closest would be general client docs |
| `3. WORK FILES (TTSZ, PICS, COMPS)` | *(no equivalent)* | No direct equivalent — aerial photo, site plan are close but wrong folder |
| `4. CLIENT BILLING (Invoice, LOE)` | *(no equivalent in Section 4)* | LOE lives in Phase 5; no billing upload in Section 4 |
| `5. LETTER OF RELIANCE (LOR)` | *(no equivalent in Section 4)* | Not in Section 4 |

**Conclusion:** Section 4's 8 upload types (Land Title Certificate, Survey Certificate/RPR, Tax Assessment Notice, Aerial Photo, Zoning Map, Flood Map, Building Permits, Site Plan) are work-file inputs — they map most naturally to `3. WORK FILES (TTSZ, PICS, COMPS)` and possibly `2. CLIENT SUPPLIED` depending on source. There is no UI rename needed in Section 4 itself. The to-build work is the file-push routing logic that correctly drops each document type into the right SharePoint subfolder when pushing outbound.

**Owner:** co-architect (routing spec) + react-specialist (push implementation)

## 11 — Production wiring / cutover

**GOAL:** 3 Entra creds stored in Vercel env; folder-create fires in production on the valtapropertyvaluations.onmicrosoft.com tenant (which IS the production environment — no account swap needed).

**Key fact:** the Entra app is registered single-tenant on the Valta tenant (`valtapropertyvaluations.onmicrosoft.com`, `ben.crowe@valta.ca`). This is already the production SharePoint site. There is no "test account vs client account" split for this integration — the registration lives and acts on the real client tenant from day one. This is unlike DocuSeal or ClickUp where there are separate test and production workspaces.

**Cutover is purely credential storage:**

1. Ben completes the Entra registration and captures the 3 values (Tenant ID + Client ID + Secret).
2. Co-architect adds 3 env vars to Vercel: `SHAREPOINT_TENANT_ID`, `SHAREPOINT_CLIENT_ID`, `SHAREPOINT_CLIENT_SECRET`.
3. Deploy the `sharepoint-folders` edge function.
4. The dashboard Client Folders section un-disables (button guard reads from env).
5. Run the Section 8 test protocol against a real test job on the live SharePoint.

**No test/prod account swap is needed** — the Valta tenant is both test and production for this integration. Use a dedicated test job number for initial testing to avoid polluting the live 2026 job folder with ghost folders.

**Secret rotation:** the client secret has a chosen expiry (24 months recommended in the recipe). Set a calendar reminder to rotate before expiry — a lapsed secret silently breaks all folder creation.
