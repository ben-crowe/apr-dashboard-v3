## 1. The "New Custom Valcre Field" column — the add/don't-add directive (Ben-flagged)

The Field Registry tab's **`New Custom Valcre Field`** column is a Yes/No that tells us, per field,
whether to **add it as a custom field in Valcre** (Yes) or **map it to an existing native Valcre
field** (No). This resolves the native-vs-custom question the dropdown audit had open. Ignore the
**`App Location`** column — client says it's just confusing.

- **No (≈22 — native, map straight through):** JobNumber, JobName, JobStatus, SubjectProperty,
  ClientCompanyName, ClientOrganizationAddress, ClientFirstName/LastName/Title/Phone/Email,
  PropertyName, PropertyAddress, PropertyType, PropertySubtype, InspectionDate, Fee, PaidDate,
  RequestDate (→ Valcre **BidDate**), SignedDate (→ Valcre **AwardDate**), DueDate, AmountPaid.
- **Yes (≈45 — ADD as custom Valcre fields):** Tenancy, StateofImprovements, StatusofImprovements,
  CurrentUseImprovements, ProposedUseImprovements, InterestAppraised, AuthorizedUse, ValueScenarios,
  ApproachestoValue, AssignmentType, ReportType, DesktopReport, Valuetimeframe, DeliveryTime, Paid,
  ClientDocuments, PreviouslyAppraised, TransactionStatus, ZoningStatus, LandMetric, CMHCFinancing,
  ValueScenario1–6, EA/HCSummary1–6, EADetail1–6, HCDetail1–6.

That's why some mapped fields go to custom rather than native Valcre — the client has now marked it explicitly.

### 1b. CUSTOM-FIELD MAPPING DOCTRINE (Ben + Chris, locked 2026-06-05)

The single most important rule for reading this registry:

- **Valcre will NOT change the dropdown options on any of its built-in (native) fields.** Native field option sets are a hard ceiling — we cannot bend them.
- **The client CAN create custom Valcre fields with his own dropdowns** — and that's exactly what the
  `New Custom Valcre Field = Yes` set is. He built (or will build) custom fields to carry the options he wants.
- **The trap:** a "Yes" field often has a name and option set that look *similar* to a native Valcre
  field, so our app may have been wired to map into the **native** field by mistake. That is WRONG — a
  near-similar name / near-similar dropdown is NOT a license to map to the native field.
- **Chris caught this live:** several of our app dropdown fields (in Client Submission and/or Job Prep)
  were pointed at the actual native Valcre dropdowns when they should have been pointed at his **custom**
  ones. Root cause: we hadn't created the custom fields yet, and didn't realize those fields were meant to be custom.

**What this means for the work:**

1. **Every `Yes` field maps to a CUSTOM Valcre field — never a native one**, even when a native field of a similar name exists.
2. **Those custom fields may not exist in Valcre yet** → there may be **no API field-ID / key** until the
   field is created. That's expected. Mark them "custom — pending creation," don't force a native ID into the slot.
3. **Action item — "Make sure these are added to Valcre as custom fields":** the `Yes` set is the creation
   checklist. QA verifies which already exist vs need creating; the rest get created (client decision per field).
4. **Audit our current app mappings** for any `Yes` field that is currently wired to a native Valcre field
   and re-point it to the custom field (or to "custom — pending" until created). Do NOT keep forcing a
   near-match native mapping.

## 4. CASCADE STATUS MISMATCH — critical bug (→ react-spec)

Our cascade code (`src/utils/loe/loeCascade.ts` `STATUS_TO_SCENARIOS`) keys on **"Existing -"** prefixes:
`Existing - Completed/Renovated/Under Renovation/To Be Renovated` + 3 `Proposed -`.

The v03 master `StatusofImprovements` options use **"Improved -"** prefixes:
**Improved - Completed · Improved - Renovated · Improved - Under Renovation · Improved - Proposed Renovation**
· Proposed - Vacant Land · Proposed - Improved Land (Demolition Required) · Proposed - Under Construction.

→ The cascade is **exact-string keyed**, so the 4 "Improved -" values will NOT resolve (only the 3
"Proposed -" match). **The cascade silently won't fire for improved properties.** Fix: realign the
cascade keys + the §5/§9/§10 derivation to the v03 status strings. (Also note client list-name typos
"Satus"/"Sate" — values are what matter, not the tab label.)
