# Asset Storage — Valta APR Dashboard Workflow

Two storage layers serve this workflow. They are NOT the same thing — one is the
client/business document home, the other is the app's image-processing pipeline.

## 1. MS SharePoint — client-facing asset storage (the business home)

The primary, client-facing asset/document storage for the Valta APR workflow lives in
**Microsoft SharePoint**. Each job has its own folder, grouped by year.

### Site / library
- **Site:** `valtapropertyvaluations.sharepoint.com/sites/V`
- **Library:** `Shared Documents`
- **Jobs root:** `2.Jobs`

### Per-job folder structure (confirmed)
```
.../Shared Documents/2.Jobs/{YEAR}/{JOB_NUMBER} - {PROPERTY_DESCRIPTION}/
```

Example (live):
```
https://valtapropertyvaluations.sharepoint.com/sites/V/Shared Documents/2.Jobs/2026/VAL261053 - Proposed 6-Storey 1804 34 Street SE Calgary AB
```

### Folder-path formula (for app wiring)
Resolve a job's SharePoint folder from values the dashboard already holds:
- `{YEAR}` — the job year (e.g. `2026`), derivable from the job number (`VAL26…` → 2026) or the job's created/effective date.
- `{JOB_NUMBER}` — the Valcre job number (e.g. `VAL261053`).
- `{PROPERTY_DESCRIPTION}` — the job's property title/address as it appears on the folder
  (e.g. `Proposed 6-Storey 1804 34 Street SE Calgary AB`).

```
BASE = https://valtapropertyvaluations.sharepoint.com/sites/V/Shared Documents/2.Jobs
folder = `${BASE}/${YEAR}/${JOB_NUMBER} - ${PROPERTY_DESCRIPTION}`
```

### The app OWNS the folder (confirmed) — deterministic, app-created
The folder name is the **job name generated by the APR app**, and the **app creates the
SharePoint folder when a job is received** (at intake). So:
- The path is **fully deterministic** — the app already knows the job number + job name it
  assigned, so it constructs the exact folder path with no search/lookup needed.
- Folder creation is a **new step in the intake chain** (alongside Valcre job create +
  ClickUp card create). On "job received," the app should create
  `2.Jobs/{YEAR}/{JOB_NUMBER} - {JOB_NAME}/` in SharePoint.
- This means the integration needs **write access** to SharePoint (create folder), not just
  read — pushing the access question (Microsoft Graph API) onto the critical path for this.

## 2. Supabase Storage — in-app image pipeline (app-side, verified)

Inside the dashboard app, image assets flow through Supabase storage buckets. Confirmed from
the codebase + migrations:

| Bucket | Purpose | Notes |
|---|---|---|
| `appraisal-raw` | Original uploaded photos land here | Public image bucket, 50 MB/file cap. Path: `uploads/{jobId}/raw/{fileId}` |
| `appraisal-processed` | Resized versions (thumb / web / print) | Public, 50 MB/file. Paths: `processed/{jobId}/thumb|web|print/{fileId}` |
| `report-images` | Appraisal report photos, by job + category | Earlier bucket, 20 MB/file cap |
| `documents` | Job documents (site survey, environmental report, etc.) | Used by the job-details upload zone |

The **most recent** storage work was the Image Configurator's `appraisal-raw` +
`appraisal-processed` buckets, set up alongside the page-layout tables that drive the
configurator.

Source: `src/features/image-configurator/types/index.ts` (`STORAGE_BUCKETS`),
`supabase/migrations/20260103_image_configurator.sql`,
`supabase/migrations/20251208_create_report_images_bucket.sql`,
`src/utils/supabaseStorage.ts`.

## How the two relate (to confirm)

Working assumption: SharePoint is the durable client/business record (app-created folder per
job); Supabase buckets are the working pipeline that the in-app image configurator + report
builder use. The hand-off between them (does anything sync SharePoint ↔ Supabase, or are they
separate?) is **open** — confirm with Ben.

## Intake-chain placement (new)

The SharePoint folder-create joins the existing on-intake automation chain:

```
Job received in APR app
  → create Valcre job
  → create ClickUp card
  → create SharePoint job folder  ← NEW: 2.Jobs/{YEAR}/{JOB_NUMBER} - {JOB_NAME}/
```

All three keyed off the same job the app just assigned. Build owner: react-spec, once the
Graph API access is set up.
