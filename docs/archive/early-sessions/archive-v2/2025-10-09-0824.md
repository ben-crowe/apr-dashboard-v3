## Key Technical Findings

**Database Architecture**
- System uses TWO tables for job data: `job_submissions` and `job_loe_details`
- VAL number MUST save to both tables for proper persistence
- Button state reads from `job_loe_details.job_number`
- List view vs detail view read from different sources

**Testing Automation Breakthrough**
- Established Marcel + Backend Agent collaboration using Playwright MCP
- Marcel automated: form filling, button clicks, console log capture, network traffic monitoring, persistence verification
- Eliminated manual testing cycles
- Generated detailed reports with exact debugging data

**ClickUp Workflow Strategy**
- Two distinct workflows identified:
  - **Workflow A**: Client form submission → Notification task on intake board (no VAL needed)
  - **Workflow B**: Manual dashboard job → Work task on main board (VAL required)
- Current button correctly requires VAL for Workflow B
- Workflow A to be implemented separately (different board, different automation)

**Session Context Management**
- Natural compression superior to manual `/compress`
- Claude Code auto-reads architecture docs during compression (491-line SESSION-SUMMARY file)
- Custom session summary command generates structured handoff docs
- Vector database (Pinecone) integration for semantic search of session history

**Valcre API Integration**
- POST-only fields exist (read-only after creation)
- PATCH updates work for most fields (contrary to old documentation)
- Property.DescriptionText used as catch-all by previous developer
- Field mapping should be direct, not through conversion tables
