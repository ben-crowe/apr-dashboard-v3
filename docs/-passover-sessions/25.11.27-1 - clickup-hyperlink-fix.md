## Problem Statement

ClickUp task descriptions were showing markdown syntax `[text](url)` as literal text instead of rendering as clickable blue hyperlinks. This affected:
- APR Dashboard job links in Stage 1 (task creation)
- Valcre job links in Stage 2 (LOE update trigger)

Links would either:
1. Show as literal markdown: `[APR Dashboard](https://url.com)`
2. Disappear entirely when Stage 2 update triggered
3. Point to wrong URL format (Valcre job links)

---

## Discovery Process

### Testing Methodology
Created test script to try 8 different link formats simultaneously on same task:
1. Markdown: `[text](url)`
2. HTML: `<a href="url">text</a>`
3. Plain text + URL: `text url`
4. URL only: `url`
5. Pipe separator: `text | url`
6. Dash separator: `text - url`
7. Parentheses: `text (url)`
8. Colon: `text: url`

**Result:** Format 1 (Markdown) showed as clickable blue "text" in ClickUp UI

### Root Causes Identified

1. **Wrong API Field**
   - Was using both `description` and `markdown_description` fields
   - ClickUp only processes markdown in `markdown_description` field
   - When both sent, ClickUp ignores markdown processing

2. **ClickUp Processing Behavior**
   - INPUT: `markdown_description: "**Label:** [text](url)"`
   - PROCESSING: ClickUp strips `**bold**` markers
   - OUTPUT: `description: "Label: text"` (clickable link, no bold)
   - READ: `markdown_description` returns NULL (write-only field)

3. **Regex Pattern Mismatch**
   - Update function looked for: `/📍 \*\*NEW APPRAISAL REQUEST:\*\*/`
   - Actual text after processing: `📍 NEW APPRAISAL REQUEST:` (no bold markers!)
   - Regex couldn't match → link not preserved/updated

4. **Wrong Valcre URL Format**
   - Was using: `https://app.valcre.com/jobs/{id}`
   - Should be: `https://app.valcre.com/job/edit/{id}#job`
   - Didn't match frontend "View Valcre Job" button URL

## Technical Implementation

### ClickUp Markdown Processing Rules

**When Writing (POST/PUT):**
```typescript
// Use markdown_description field
{
  "markdown_description": "**Label:** [Link Text](https://url.com)"
}
```

**What ClickUp Does:**
1. Strips `**bold**` markers → plain text
2. Converts `[text](url)` → clickable hyperlink
3. Stores processed result in `description` field
4. `markdown_description` remains NULL when reading

**When Reading (GET):**
```typescript
// Read from description field
const task = await fetch('/api/v2/task/{id}')
const text = task.description  // Contains processed text
// task.markdown_description is NULL!
```

### Update Function Pattern

```typescript
// 1. Fetch existing task
const existingTask = await getTask(taskId)
const existingDesc = existingTask.description  // Read from description!

// 2. Extract sections
const stage1 = existingDesc.match(/before LOE section/)

// 3. Update with markdown (NO bold markers in regex!)
const updated = stage1.replace(
  /📍 LABEL:/,  // Match processed text (no **)
  `📍 **LABEL:** [Link](${url})`  // Send with markdown
)

// 4. Send to markdown_description field
await updateTask(taskId, {
  markdown_description: updated  // Write to markdown_description!
})
```

## Key Learnings

### 1. ClickUp API Field Behavior
- `markdown_description` = INPUT field (write-only, processes markdown)
- `description` = OUTPUT field (read-only, contains processed text)
- Sending to both fields causes ClickUp to skip markdown processing

### 2. Markdown Processing
- ClickUp strips **bold** markers automatically
- Converts `[text](url)` to clickable links
- Raw description shows just "text" (no brackets/URL)
- Links only clickable in UI, not visible in raw text

### 3. Regex Matching Strategy
- Match against PROCESSED text (after ClickUp processing)
- Don't look for bold markers (they're stripped)
- Don't look for markdown syntax (it's converted)
- Match the plain text that remains

### 4. URL Format Validation
- Always check frontend button/link for correct URL format
- Cross-reference Edge Function URLs with UI URLs
- Don't assume URL patterns - verify actual working links
- Include URL fragments (#job, #tab, etc.) if needed

### 5. CLI Script Library Value
- Tested scripts = proven patterns
- Removes guesswork from future implementations
- Documents "how to do X" with working examples
- Enables rapid testing before production deployment
