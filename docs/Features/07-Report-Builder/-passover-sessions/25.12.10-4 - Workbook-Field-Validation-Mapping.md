### 11-Step Calculation Flow

```typescript
1. Unit Mix → Rental Income by type (1BR, 2BR, 3BR, 4BR)
2. Other Income (parking, laundry, storage, etc.)
3. PGR (Potential Gross Revenue) = Rental Income + Other Income
4. Vacancy Loss = PGR × vacancy rate %
5. EGR (Effective Gross Revenue) = PGR - Vacancy Loss
6-7. Operating Expenses (5 calculation bases):
    - per_unit: value × total units
    - per_sf: value × total square feet
    - percent_egr: EGR × percentage
    - percent_pgr: PGR × percentage
    - fixed: exact dollar amount
8. NOI (Net Operating Income) = EGR - Total Operating Expenses
9. Indicated Value = NOI ÷ Cap Rate
10. Round to nearest $10,000
11. Post-value adjustments (if any)
```

**Validation Result:** All 7 metrics exact match to Valcre workbook
- Final indicated value: **$1,780,000** (exact)
- PGR, EGR, NOI, Expenses - all validated ✅

## 🚨 Critical Discovery - THREE Naming Conventions

**The Root Cause of Field Mapping Issues:**

| Source | Example | Convention | Count |
|--------|---------|------------|-------|
| **Valcre Workbook** | `IA_DirCap_NOI` | PascalCase_Prefix | 7,988 named ranges |
| **APR Registry** | `calc-noi` | kebab-case | 392 fields |
| **PDF Visual** | `income_noi` | snake_case | Variable content |

**Problem Identified:**
- Registry was built independently without referencing workbook naming
- NO mapping layer existed to connect workbook data → registry fields
- Workbook import impossible without translation
- Field names were intentionally cleaned from messy Valcre conventions

**Why This Matters:**
```
Valcre Workbook Export
  ↓
  ? (NO MAPPING)
  ↓
APR Dashboard Registry
  ↓
Template Rendering
  ↓
PDF Output
```

**Impact:**
- Workbook data cannot populate dashboard fields
- Manual data entry required (defeats automation purpose)
- Operational V3 Dashboard can't leverage Valcre workbooks

## ✅ Solution - Translation Layer (NOT Refactor)

### Design Decision: Keep Clean Names, Add Metadata

**Approach:**
```typescript
// BEFORE (no connection)
{
  id: 'calc-noi',
  storeId: 'calc-noi',
  type: 'currency',
  section: 'calculator'
}

// AFTER (explicit mapping)
{
  id: 'calc-noi',                    // Clean kebab-case (unchanged)
  workbookRange: 'IA_DirCap_NOI',    // Valcre name (new)
  storeId: 'calc-noi',
  type: 'currency',
  section: 'calculator'
}
```

**Why This Works:**
1. ✅ **Non-Breaking:** Existing field IDs unchanged (all templates still work)
2. ✅ **Explicit:** Translation is documented, not inferred
3. ✅ **Discoverable:** Helper functions make mapping easy to use
4. ✅ **Maintainable:** unmappedWorkbookFields tracks what's still missing
5. ✅ **Scalable:** Adding new mappings is straightforward

**Workbook Import Flow:**
```
1. Read 'IA_DirCap_NOI' from Valcre workbook
   ↓
2. Look up mapping: getRegistryId('IA_DirCap_NOI')
   ↓
3. Returns: 'calc-noi'
   ↓
4. Populate store field: store['calc-noi'] = value
   ↓
5. Template renders: ${calcNoi}
   ↓
6. PDF displays calculated value
```

## 🔧 Technical Implementation

### Workbook Import Flow (Planned)

```typescript
import { getRegistryId } from './schema/workbookFieldMapping';
import { store } from './store/reportBuilderStore';

// 1. Read workbook (Excel.js or similar)
const workbook = await readWorkbook('VAL251012.xlsm');

// 2. Extract named ranges
const namedRanges = workbook.getNamedRanges();

// 3. Map to registry and populate store
namedRanges.forEach(range => {
  const workbookName = range.name;
  const value = range.value;

  // Look up registry field ID
  const registryId = getRegistryId(workbookName);

  if (registryId) {
    // Populate store field
    store.setFieldValue(registryId, value);
  } else {
    // Log unmapped field for gap tracking
    console.warn(`Unmapped workbook field: ${workbookName}`);
  }
});

// 4. Trigger template re-render
store.notifyUpdate();
```

## 🎓 Key Learnings

### Why Renaming Was Correct

**Valcre Naming Issues:**
- Inconsistent: `IA_DirCap_NOI` vs `Subject_NRA` vs `Value_FinalScenario1`
- Hard to read: `IA_DirCap_Expense_RM_EGR`
- No pattern: Mix of prefixes, abbreviations, casing

**Registry Naming Benefits:**
- Consistent: `calc-noi`, `total-nra`, `recon-final-value`
- Readable: `calc-exp-repairs`, `rent-1br-concluded`
- Modern convention: kebab-case standard in React/TypeScript

**The Problem:** Not the rename, but **forgetting to document the translation**

### Translation Layer Pattern

**Key Insight:** You don't need to change names to connect systems

**Pattern:**
1. Keep clean, modern field IDs in registry
2. Add `workbookRange` metadata property
3. Build translation object (workbookToRegistry)
4. Use helper functions for lookups
5. Track gaps explicitly (unmappedWorkbookFields)

**Benefits:**
- Non-breaking to existing code
- Explicit (no inference, no guessing)
- Maintainable (single source of truth)
- Discoverable (helper functions, statistics)

### Gap Tracking Best Practice

**Before:** Manual spreadsheets, unclear what's missing

**After:** Programmatic tracking
```typescript
// Explicit list of what's NOT mapped
export const unmappedWorkbookFields: string[] = [
  'Sale4_Name',
  'IA_DirCap_NOIPSF',
  // ... 50+ items
];

// Statistics function
getMappingStats() → {
  mapped: 252,
  unmapped: 50,
  coveragePercent: 83
}
```

**Benefits:**
- Always know coverage percentage
