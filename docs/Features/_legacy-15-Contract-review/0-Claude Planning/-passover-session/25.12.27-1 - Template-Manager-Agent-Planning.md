## ✅ Decisions Made

### 1. Create Hybrid Agent
**Decision:** Build `apr-frontend-template-expert` combining:
- Frontend developer capabilities (edit pages, HTML/CSS, styling)
- Template manager expertise (version control, field validation, git commits)
- Desktop handoff workflow (export/import pages seamlessly)

**Why:** Claude Desktop can't read codebase, validate fields, or commit changes

### 2. Use Existing Workspace
**Decision:** Use `/docs/15-Contract-review/1-Formatting-template/` as agent workspace
**Why:** Folder already has individual pages, field mappings, and historical versions

### 3. Bidirectional Handoff
**Decision:** Enable workflow where either Claude Desktop or Claude Code can:
- Edit pages
- Hand off to other tool
- Continue where other left off

**Benefit:** Best of both worlds - Desktop visual preview + Code codebase integration

### 4. Version Tracking Method
**Decision:** Keep version in HTML comment header (already exists!)
**Implementation:**
- VERSION line at top of file (line 7)
- CHANGELOG after line 19
- Update both when merging/bumping version

## 🎓 Key Learnings

### 1. 4-File Sync System is Critical
- fieldRegistry.ts = source of truth
- Template placeholders must match registry IDs exactly
- Test data keys must match registry IDs
- Image mappings must match registry IDs
- **If ANY mismatch** = broken rendering

### 2. Template System Architecture
- 77 unique pages × 2 page-sheets = 250 total divs
- Each page scoped with `.page-XX` CSS prefix
- Placeholders use `{{field-id}}` format (kebab-case)
- Template loaded via fetch() from `/public/` (hardcoded path)

### 3. Field Naming Convention
- ALWAYS kebab-case: `comp1-sale-price`
- NEVER camelCase or snake_case
- Prefixes indicate category: calc-, comp1-5, img-, site-, etc.
- 1,687 total fields across 13 sections

### 4. Version Tracking Works
- HTML comment header is perfect for version info
- Changelog can list all changes per version
- Synced files (registry, test data) should have matching version
