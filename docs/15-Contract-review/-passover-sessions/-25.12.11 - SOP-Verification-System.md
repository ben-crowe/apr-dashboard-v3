## 🚨 CRITICAL PRINCIPLES (Discovered 2025-12-11)

### Principle 1: Replicate the Reference - Don't Ask "Should We?"

**Problem:** Agent asked: "Should we add YTD 2025 data to the table, or use PROJECTION values for both columns?"

**Issue:** The reference report ALREADY SHOWS two columns with different values (YTD vs PROJECTION). This question should never have been asked.

**Correct Approach:**
1. ✅ Look at reference report → See two columns with different data
2. ✅ Conclusion: Obviously the original appraiser needed both value sets
3. ✅ Action: Go to Valcre workbook and find WHERE both value sets come from
4. ✅ Replicate: Map both value sets to our system exactly as reference shows

**Rule:** If the reference report has it, we build it. Period. Don't decide whether to build something different - replicate what the reference shows.

---

### Principle 2: Research the Source - Don't Ask Questions

**Problem:** Agent asked questions that could be answered by checking source documents.

**Issue:** We have all the source documents:
- ✅ Valcre Workbook (VAL251012.xlsm) - THE SOURCE OF TRUTH
- ✅ Reference Page Images (79 PNGs)
- ✅ Reference HTML
- ✅ Existing documentation files

**Correct Approach:**
1. ✅ Agent has a question → Check sources FIRST
2. ✅ Open the Valcre workbook → Find where data comes from
3. ✅ Read the reference images → Extract actual values
4. ✅ Check existing docs → See what was already mapped
5. ❌ Don't ask - Research and answer your own question

**Rule:** Before asking any question, exhaust all source documents first. The answer is almost always in the sources.

### Cross-Referencing Protocol (How to Research)

**When building any component, follow this sequence:**

1. **Check Reference Image**
   - `/Ref REPORT Page Images/Page X of 79.png`
   - What does the actual output look like?
   - What data is displayed?

2. **Check Valcre Workbook**
   - Open `VAL251012.xlsm`
   - Where does that data come from?
   - What are the named ranges/cells?

3. **Check Existing Documentation**
   - PAGE-INVENTORY.md (page structure)
   - Existing field mapping docs
   - fieldRegistry.ts (already defined fields)

4. **ONLY THEN** - If sources don't have the answer, ask for clarification

**This prevents:**
- ❌ Asking "should we?" when reference already shows the answer
- ❌ Creating synthetic test data when real data exists in workbook
- ❌ Duplicating work that was already documented

### Systematic Verification Workflow (The Complete Process)

**When agent asks: "Should we add X?" or "What should we do about Y?"**

**STOP. Follow this workflow instead:**

```
Step 1: Check Extracted Test Data
   └─ What do we have in northBattlefordTestData-REAL.ts?
   └─ Does the field exist? (calc-noi, calc-pgr, etc.)
   └─ Does it have a value?

Step 2: If Missing → Go to Reference Page
   └─ Open reference image: /Ref REPORT Page Images/Page X of 79.png
   └─ Extract EVERY value from the table
   └─ Document every cell, every sub-column, every total

Step 3: Compare to Test Data
   └─ What exists in test data? ✅
   └─ What's missing from test data? ❌
   └─ Create gap list

Step 4: For Missing Values → Go to Workbook
   └─ Open VAL251012.xlsm
   └─ Find where that value comes from (named range, cell reference)
   └─ Document the source

Step 5: Repeat Until EVERY Item Verified
   └─ Every field has a value in test data ✅
   └─ Every value traced back to workbook source ✅
   └─ No gaps remaining ✅
```

**Example - Page 44 "Operating History" Table:**

1. ✅ Check test data → Has calc-pgr: 204240, calc-noi: 111771
2. ✅ Check reference Page 44 → Shows TWO columns (YTD 2025 vs PROJECTION)
3. ❌ Compare → Test data only has PROJECTION values, YTD values MISSING
4. ✅ Go to workbook → Find where YTD 2025 values come from
5. ✅ Document → Add YTD fields to test data with workbook source

**Don't ask "should we add YTD?" - The workflow tells you: Yes, because it's missing and reference shows it.**

**What Actually Happened:**

❌ **Agent suggested alternatives** instead of replicating what reference shows

**What Agent SHOULD Have Done:**

1. ✅ **Principle 1 - Replicate the Reference:**
   - Reference shows TWO columns with DIFFERENT values
   - Therefore: Both value sets exist and are needed
   - Action: Replicate exactly what reference shows

2. ✅ **Principle 2 - Research the Source:**
   - Open Valcre workbook (VAL251012.xlsm)
   - Search for YTD or historical data sections

3. ✅ **Understanding the Data:**
   - **PROJECTION column** = Our calc engine (already works)

   - These are likely template calculations, not calc engine

**The Correct Answer (No Question Needed):**

The reference shows TWO columns with DIFFERENT numbers. Therefore:
1. Go to Valcre workbook
2. Find the YTD 2025 data source
3. Map those fields to registry (separate from PROJECTION fields)
4. Build the table with BOTH columns as shown in reference

**Key Learning:** Never ask "should we build X?" if the reference already has X. The reference IS the specification.

**Additional Context (User Clarification):**

When user said "calc engine is not to be touched," they were confirming:
- ✅ Calc engine generates PROJECTION values (already validated)
- ✅ This is about DATA PLACEMENT for display only
- ✅ YTD values are separate INPUT fields, not calc engine outputs

## 🎓 KEY LEARNINGS

### 1. Documented Workflow Pattern is Essential
Following structured workflow (PAGE-INVENTORY → LAYOUT-PATTERN → COMPONENT-CATALOG → Build) prevents errors and ensures consistency. Don't build blindly - always consult documentation suite first.

### 2. Pre-Integration Verification Highly Effective
Systematic verification before building catches missing prerequisites and builds confidence. Always verify fields exist before building components.

### 3. Reference Image Verification is Critical
ALWAYS verify reference images BEFORE building. Don't assume from task specs. Two minutes of verification prevents hours of rework.

### 4. Use ACTUAL Reference Data for Testing
Extract real values from reference images, not synthetic test data. Real data catches structural issues synthetic tests miss.

### 5. Replicate the Reference - Don't Ask "Should We?"
If the reference has it, we build it. Period. The reference IS the specification. Don't decide to build something different.

### 6. Research the Source - Don't Ask Questions
Before asking any question, exhaust all source documents first. The answer is almost always in the sources (workbook, reference images, existing docs).

### 7. Systematic Verification Workflow Works
The 5-step process (check test data → reference page → compare → workbook → repeat) ensures nothing is missed and prevents "should we?" questions.
