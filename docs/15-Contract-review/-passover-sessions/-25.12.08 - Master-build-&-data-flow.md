## 🚨 CRITICAL MENTAL MODEL SHIFT - READ THIS FIRST

### The Builder Is NOT a Flowing HTML Document

**WRONG THINKING:**
```
"Dynamic HTML that flows and breaks into pages"
"Pagination system to split content across pages"  
"Page numbers and page breaks"
```

**CORRECT THINKING:**
```
"Component-based page builder (like Squarespace/Webflow)"
"Discrete, predefined section templates"
"Fixed layouts with content slots"
```

## 🎯 THE REAL PROBLEM (NOT Pagination)

### What We Thought Was Wrong
- "Pagination code is broken"
- "Page numbers not showing"
- "Need to fix page breaks"

### What Was ACTUALLY Wrong
- **Missing 45 pages of CONTENT**
- Not rendering all section instances
- Skipping content during render
- 25 pages output vs 70 pages reference = **MISSING CONTENT**, not pagination issue

### The Fix: Component Templates

**Before (Dynamic HTML):**
```typescript
// Dynamic rendering - content flows wherever
<div className="photos">
  {photos.map(p => <img src={p} />)}
</div>
// Result: Unpredictable layout, missing sections
```

**After (Component Templates):**
```typescript
// Fixed section template - always same structure
function PhotoSection({ images }: { images: [string, string, string, string] }) {
  return (
    <div className="photo-section">
      <img src={images[0]} className="photo-slot-1" />
      <img src={images[1]} className="photo-slot-2" />
      <img src={images[2]} className="photo-slot-3" />
      <img src={images[3]} className="photo-slot-4" />
    </div>
  );
}

// If data has 24 photos → Render 6 PhotoSection components
// If data has 12 photos → Render 3 PhotoSection components
// NOTHING gets skipped
```

```typescript
// Photo Section Component
interface PhotoSectionProps {
  images: [string, string, string, string];
  captions?: [string, string, string, string];

export function PhotoSection({ images, captions }: PhotoSectionProps) {
  return (
    <section className="photo-section">
      <div className="photo-grid">
        <div className="photo-item">

          {captions?.[1] && <p className="caption">{captions[1]}</p>}
```

### Step 3: Update reportHtmlTemplate.ts

**Goal:** Replace dynamic renderers with component-based templates

**Current Problem:**
```typescript
// Dynamic rendering - unpredictable output
function renderPhotosSection(data) {
  return data.photos.map(p => `<img src="${p}" />`).join('');
}
```

**Solution:**
```typescript
// Component-based - predictable output
function renderPhotosSection(data) {
  const photoGroups = chunk(data.photos, 4); // Group into sets of 4
  return photoGroups.map((group, i) => 
    PhotoSectionTemplate({
      images: group,
      sectionNumber: i + 1
    })
  ).join('');
}

function PhotoSectionTemplate({ images, sectionNumber }) {
  return `
    <section class="photo-section">
      <h3>Property Photos - Set ${sectionNumber}</h3>
      <div class="photo-grid">
        ${images.map((img, i) => `
          <div class="photo-item">
            <img src="${img}" alt="Photo ${i + 1}" />
          </div>
        `).join('')}
      </div>
    </section>
  `;
}
```

**Engine Location:**
- **File:** `src/features/report-builder/store/reportBuilderStore.ts`
- **Function:** `runCalculations()` (lines 3181-3374, 193 lines)
- **Architecture:** 11-step sequential calculation flow with 5 expense calculation bases

**11-Step Calculation Flow:**
1. Unit Mix Calculations (rental income by unit type)
2. Other Income (parking, laundry, etc.)
3. Potential Gross Revenue (PGR = Rental + Other)
4. Vacancy & Collection Loss (PGR × vacancy rate)
5. Effective Gross Revenue (EGR = PGR - Vacancy)
6-7. Operating Expense Calculations (5 different bases)
8. Net Operating Income (NOI = EGR - OpEx)
9. Cap Rate Valuation (Value = NOI ÷ Cap Rate)
10. Rounding to $10,000 (industry standard)
11. Post-Value Adjustments (extraordinary items)

**5 Expense Calculation Bases:**
1. **Per Unit** (`per_unit`) - Value × Units (e.g., taxes, insurance)
2. **Per Square Foot** (`per_sf`) - Value × SF (area-based expenses)
3. **Percentage of EGR** (`percent_egr`) - EGR × % (management fees)
4. **Percentage of PGR** (`percent_pgr`) - PGR × % (rare)
5. **Fixed Dollar Amount** (`fixed`) - Flat rate

**Critical Discovery: Management Fees MUST Use EGR**

**Issue:** Management fees were initially calculated using PGR instead of EGR.

**Why This Matters:**
- Management companies are compensated on **collected revenue** (EGR), not potential revenue (PGR)
- Using PGR overstates management costs and undervalues the property

**Financial Impact Example:**
```
WRONG (using PGR):
Management = 4.25% × $204,240 = $8,680

CORRECT (using EGR):
Management = 4.25% × $196,070 = $8,333

Difference: $347 per year
Impact on Value: ~$5,552 at 6.25% cap rate
```

**Resolution:** Engine updated to use EGR for `percent_egr` expense base, achieving exact Valcre alignment.
